# 🦒 Gironimo Website Specification

---

# Deployment Constraints

* Static site only. Deploy to GitHub Pages.
* No backend, database, or runtime external API dependencies.
* Must function in standard GitHub Pages environment.
* All functionality must run client-side.

---

# File Locations

* Implementation root: `/docs/<model-name>/`
* CSV source file: `/docs/results/leaderboard.csv`
* Runtime CSV URL: `/results/leaderboard.csv`

The leaderboard MUST be fetched at runtime using `/results/leaderboard.csv`.

---

# Rule Priority

If constraints conflict:

1. Integration rules (state + system behavior)
2. Feature rules
3. Design rules

---

# Core System Requirement

All application state MUST be managed via a **single centralized store**.

The store is the only source of truth.

---

## Store Requirements

* Single global state object
* Reducer-based updates (action → next state)
* Subscription-based UI updates
* No duplicated state outside store

---

## Atomic Update Rule

All state updates within a tick must resolve into a single render.

* No intermediate render states
* Exactly one UI commit per tick

---

## Required State Fields

* `theme`
* `selectedPhilosophyStatements`
* `selectedModel`
* `workflowState`
* `leaderboardData`

---

## Derived State Rule

Derived values MUST NOT be stored.

Includes:

* sorting
* filtering
* grouping
* ranking
* UI computation flags

All derived data must be computed during render.

---

## Persistence

* Must use `localStorage`
* Must handle missing/corrupt state safely

---

## Isolation Rule

Features MUST NOT:

* access each other’s internals
* mutate external DOM outside render flow
* bypass store actions

All interactions go through store events.

---

# Design System Rules

* Fully responsive design required
* WCAG AA compliance required
* Keyboard accessible required

### Design Freedom (Important)

Within accessibility constraints:

* color system is unrestricted
* typography is unrestricted
* spacing system is unrestricted
* animation system is unrestricted

Design must remain:

* consistent
* readable
* usable

No aesthetic style is enforced.

Must include:

* Button (all states)
* Card
* Modal
* Tooltip
* Notification
* ThemeProvider

Rules:

* reusable components only
* no feature-specific logic
* props-driven configuration

# Feature 2: Philosophy Explorer

Statements:

| Category      | Statement                                                                    |
| ------------- | ---------------------------------------------------------------------------- |
| Practical     | Five minutes of correct work beats thirty seconds of plausible garbage       |
| Practical     | Human gates at specification and architecture catch expensive mistakes early |
| Philosophical | AI amplifies human judgment; it doesn't replace it                           |
| Philosophical | Stand tall. See far. Move deliberately                                       |
| Technical     | Two-model critique catches what single-model confidence misses               |
| Technical     | Documentation is auto-drafted because it fails when it's optional            |
| Technical     | Own your infrastructure. Own your data                                       |

Rules:

* multi-select
* category filtering
* store-persisted selection only
* selection affects system only via derived state

---

# Feature 3: Interactive Mascot (Giraffe)

Always visible.

Driven entirely by store state.

Behavior mapping:

| Condition           | Behavior                            |
| ------------------- | ----------------------------------- |
| Idle                | subtle motion                       |
| Philosophy selected | orients toward selected statements  |
| Theme change        | visual transition reaction          |
| Model selected      | orients toward leaderboard position |
| Workflow hover      | responds directionally              |

Must be implemented using:

* state machine OR behavior tree
* no DOM-driven logic

---

# Feature 4: Workflow System

Stages:

Spec → Gate → Architecture → Gate → Implementation → Review → ADR

---

## Progress Rules

Based on `selectedModel.score`:

* ≥ 90 → full progression
* 70–89 → stops at Implementation
* < 70 → stops at Architecture

---

## Interaction Rules

* clicking stage opens modal
* modal shows:

  * selected model metadata
  * philosophy context (if selected)

---

# Feature 5: Leaderboard (CSV-driven)

## Data Source

Must fetch:

`/results/leaderboard.csv`

No hardcoded data allowed.

Must fail gracefully.

---

## CSV Format

model,date,video_url,result_url,speed,one_shot_attempts,design,architecture,code_quality,feature_complete,performance,accessibility,best_practices,value,overall_score

Invalid rows ignored.

Missing values → 0.

---

## UI Requirements

* sortable by score / attempts / date
* row selection persists
* selected model highlighted
* top performer visually emphasized

---

## System Effects

Model selection affects:

* workflow state
* mascot behavior
* philosophy highlighting

---

## Philosophy Highlighting

Triggered only when philosophy selected:

* Practical → code_quality + feature_complete ≥ 8
* Philosophical → architecture ≥ 8
* Technical → best_practices ≥ 8

---

# Cross-System Integration Rule

All cross-feature behavior MUST occur via store state only.

No direct feature-to-feature communication.

---

## Trigger → Effect Map

| Trigger              | Effect                                 |
| -------------------- | -------------------------------------- |
| Philosophy selection | leaderboard + workflow + mascot update |
| Model selection      | workflow + mascot update               |
| Theme change         | global UI update                       |
| Stage click          | modal opens                            |

---

# Performance Requirements

* 20+ models supported smoothly
* instant sorting/filtering
* no blocking UI operations

* corrupted CSV

* corrupted localStorage
* rapid state updates

* triggered by model selection

* cross-feature DOM manipulation
* hardcoded CSV data
* broken interactivity
* missing error handling
* incomplete implementation
