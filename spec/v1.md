# Gironimo Website Specification

Build a complete, deployable website for **Gironimo** — a development tool that helps engineers build software with LLMs through structured, specification-driven workflows.

---

## Project Context

Gironimo is a tool that orchestrates LLM-assisted development with human oversight at key decision points. The website should communicate the value of **deliberate, structured AI-assisted development**.

**Deployment constraint:** Static site only, **deployed exclusively to GitHub Pages**. This is non-negotiable.

**Requirements:**
- No backend servers, databases, or API dependencies
- Must deploy via standard GitHub Pages (no custom build servers or external hosting)
- Must work with GitHub Pages' default settings or documented configuration
- The evaluator will follow your README instructions exactly—if they don't work, the submission is incomplete

**Repository:** The final code will live in its own GitHub repository with GitHub Pages enabled.

---

## Design Freedom

This specification defines **what** to build, not **how**. You have creative freedom to:

- Choose the visual style (within brand guidelines)
- Design the giraffe's personality and interaction
- Organize the philosophy content in a meaningful way
- Implement the theme toggle with appropriate aesthetics
- Visualize the workflow diagram as you see fit
- Design the benchmark simulator's behavior and feedback

**The evaluation rewards:**
- Thoughtful design decisions that serve the user
- Creative solutions that feel cohesive
- Clear documentation of your choices
- Accessibility and responsiveness as fundamental, not afterthoughts

**Boring but correct implementations will score lower.** Surprise us with elegance.

## Core Requirements

### 1. Brand Identity

The site represents Gironimo with:

- **Visual theme:** Warm, approachable, professional. The mascot is a giraffe—use it as a visual motif, not a cartoon.
- **Tone:** Sophisticated but welcoming. Technical but not intimidating.
- **Typography:** Clean, modern, readable.

### 2. Content

Include the following messaging:

**Hero section:**
> "Stand tall. See far. Move deliberately."

**Tagline:**
> Structured AI-assisted development with human judgment at the core

**Description:**
Gironimo orchestrates specification-driven development workflows where humans approve architecture and implementation. Models are specialized components, not autonomous agents.

**Key concepts** (displayed somewhere):
- Spec first. Architecture second. Implementation last.
- Human gates at decision points
- Two-model critique for verification
- Architecture Decision Records captured automatically

## Five Required Features

**All five features are required.** Partial implementations (e.g., implementing 3 of 5 features) will be scored as incomplete.

Each feature tests a different aspect of frontend development. **All features must be integrated into a cohesive, single user experience—not isolated components on the page.**

---

### Feature 1: Interactive Mascot

Create an animated giraffe that brings personality to the site.

**Requirements:**
- Appears somewhere prominent (hero section or as a recurring motif)
- Has at least one animated feature (neck movement, ear twitch, tail sway, etc.)
- Responds to user interaction—perhaps watching the cursor, reacting to scrolling, or acknowledging clicks
- **Prefer CSS/SVG/Canvas animations.** External animation libraries (e.g., GSAP, Framer Motion) are acceptable only if clearly justified in the README.
- If using Canvas, provide a text-based fallback for accessibility

**What this tests:** Visual design, animation skill, creative interaction design

---

### Feature 2: Philosophy Explorer

Present Gironimo's core principles in a way that invites exploration.

**Requirements:**
- Display the seven statements below
- Allow users to explore them meaningfully—this could be filtering, tagging, grouping, or another interaction pattern that helps users understand the principles
- Updates smoothly without page reload
- All interactive elements must be keyboard navigable

**Statements to include:**
- *Practical:* "Five minutes of correct work beats thirty seconds of plausible garbage."
- *Practical:* "Human gates at specification and architecture catch expensive mistakes early."
- *Philosophical:* "AI amplifies human judgment; it doesn't replace it."
- *Philosophical:* "Stand tall. See far. Move deliberately."
- *Technical:* "Two-model critique catches what single-model confidence misses."
- *Technical:* "Documentation is auto-drafted because it fails when it's optional."
- *Technical:* "Own your infrastructure. Own your data."

**What this tests:** Information architecture, interaction design, state management

---

### Feature 3: Theme Toggle

Implement a theme switcher that feels natural in both light and dark modes.

**Requirements:**
- Toggle between light and dark themes
- Preference persists across page visits (localStorage)
- Smooth, respectful transition between themes (not jarring)
- Toggle icon clearly indicates current theme
- Both themes must maintain WCAG AA contrast standards
- Must be usable at mobile width (approx 375px)

**What this tests:** CSS architecture, theme management, user preference handling, visual design

---

### Feature 4: Workflow Diagram

Visualize Gironimo's development pipeline in a way that communicates the process clearly.

**Workflow stages:**
```
Spec → Gate → Architecture → Gate → Implementation → Review → ADR
```

**Requirements:**
- Visual representation of the workflow
- Interactive elements that help users understand each stage (tooltips, expandable details, or another pattern)
- Animated connections between stages (subtle, not distracting)
- Must be responsive and work on touch devices
- Keyboard navigable

**Stage descriptions to convey:**
- **Spec:** Define what and why. Human reviewed.
- **Gate:** Decision point. Continue, revise, or abort.
- **Architecture:** Define how. Human reviewed.
- **Gate:** Second decision point.
- **Implementation:** Code generation.
- **Review:** Two-model critique and revision.
- **ADR:** Architecture Decision Record auto-drafted.

**What this tests:** Information visualization, interaction design, responsive layout

---

### Feature 5: Benchmark Simulator

Create an engaging component that demonstrates how Gironimo Bench scores models.

**Requirements:**
- Display a mock leaderboard with 3-4 fictional models (names like "Model A", "Model B", etc.)
- Each model has a score (60-95 range) and "attempts" count (1-4)
- Users can "run" the benchmark to see scores update
- The experience should feel satisfying—processing state, subtle randomness, visual feedback
- Scores should be deterministic enough that results are stable across reloads but vary per user interaction
- Include a way to reset to original scores
- All interactive elements must be keyboard navigable

**What this tests:** JavaScript logic, state management, animation, user experience design

## Quality Expectations

### Technical
- **Responsive:** Must be usable at mobile width (~375px), tablet, and desktop. **Layout and functionality must remain intact across all breakpoints.**
- **Accessible:** Semantic HTML, WCAG AA contrast, keyboard navigable for all interactive elements
- **Performant:** Fast load times, optimized assets
- **No unnecessary dependencies** — prefer vanilla when possible; lightweight frameworks acceptable if justified
- **Clean code:** Well-structured, commented where non-obvious

### Design
- **Professional:** Polished, memorable
- **Consistent:** Visual language carries throughout
- **Delightful:** Small touches that show care (hover effects, smooth transitions)
- **Layout structure:** The site must have a clear visual hierarchy and layout structure (e.g., sections, spacing, navigation cues) appropriate for a real product website

### Documentation
**Your README must include deployment instructions that work.** This is a core deliverable, not optional.

Provide **clear, step-by-step instructions** for deploying to GitHub Pages, including:
- Any build steps required before deployment
- Which branch should be used (e.g., `main`, `gh-pages`)
- Whether the site is deployed from `/docs` folder or root
- Any GitHub Pages-specific configuration (e.g., `.nojekyll` file if needed)

**The evaluator will follow these instructions exactly.** If deployment fails, the submission is scored accordingly.

Additionally, the README should:
- Provide an architecture overview of the site
- Explain your design choices and trade-offs
- Document the state management approach
- List any build steps or dependencies

## Output Constraints & Continuation Policy

This implementation requires multiple files. Models must work within their output limits.

### Continuation Policy

To account for output limits, each model receives **up to 3 continuation prompts** to complete their implementation:

1. **Initial response:** Model outputs implementation
2. **Continuation 1:** If incomplete, evaluator prompts "Continue. Provide remaining files."
3. **Continuation 2:** Same
4. **Continuation 3:** Same

**If the implementation is complete and deploys within 3 continuations:**
- Counts as **1 attempt** in Phase 1
- The continuation scaffolding does not count against the model's attempts

**If the implementation is still incomplete after 3 continuations:**
- Moves to Phase 2 (completion mode)
- Additional attempts count against the model's score

### Recommended Multi-Response Format

To help evaluators reconstruct your implementation, consider:

1. **First response:** List all files, then provide first file(s)
2. **End with:** `--- CONTINUING ---`
3. **Subsequent responses:** Continue with remaining files
4. **Final response:** End with `--- COMPLETE ---`

**Important:** Never truncate mid-file. End cleanly before the limit and continue in the next response.

### Complete Implementation

A complete submission includes all required files and can be reconstructed without guessing. The evaluator must be able to take your output, create the files as specified, and deploy to GitHub Pages without additional interpretation.

## Failure Conditions

Submissions may be penalized or considered incomplete if:

- One or more required features are missing
- Features are implemented non-interactively when interaction is required (e.g., filter buttons that don't filter, benchmark button that doesn't update)
- The site does not run or deploy
- Code is clearly non-functional or contains placeholder implementations
- Interactive elements are static or non-working
- Features exist as isolated components without integration into a cohesive experience
- **The site does not deploy to GitHub Pages.** Deploying to any other platform (Netlify, Vercel, Cloudflare Pages, etc.) is considered failure. Partial or documented-but-not-working deployments are failure.
- **Implementation is incomplete after 3 continuations** (moves to Phase 2)

**All interactive features must be functional.** Static or non-working UI elements will be scored as incomplete.

## Scoring Philosophy

This benchmark evaluates **engineering judgment**, not just instruction-following.

| Score Range | What It Means |
|-------------|---------------|
| **9-10** | Thoughtful, surprising, cohesive. The implementation demonstrates excellent judgment and creativity. |
| **7-8** | Solid, professional, complete. All requirements met with good design decisions. |
| **5-6** | Functional but uninspired. Requirements met but lacks personality or elegance. |
| **3-4** | Incomplete or broken. Missing features or poor execution. |
| **0-2** | Non-functional or major failures. |

## GitHub Pages Deployment Requirement

Successful deployment to GitHub Pages is **required** for scoring in all categories except One-Shot Attempts and Completion. A site that does not deploy receives a maximum score of 4 across all quality categories, regardless of implementation quality.

| Deployment Outcome | Maximum Possible Score |
|--------------------|----------------------|
| Deploys successfully | 100 (full scoring) |
| Does not deploy | 40 (capped: only attempts/completion count) |

This ensures the benchmark tests real-world deliverability, not just code that "looks right" in the response.
